UI Examples


Example 1: Blender

Purpose: Blender is a 3D modeling and rendering package.

The good:

  • Consistent controls. Moving, scaling, and rotating are always the same keys. This applies to vertices, edges, faces, keyframes, video frames, etc.
  • When selecting an object, arrows appear that can be dragged to move it. This is one of the first functionalities a beginner learns, and it appears automatically. Unfortunately, it does not imply to a beginner the more effective means of moving, i.e. the keyboard shortcut.
  • Updating an object in one view updates it in all views. This is especially useful when painting, which can be done in 2D or 3D. When painting in 3D, the 2D view only updates after the stroke is completed (i.e. the user lets go of the mouse), while painting in 2D instantly updates in 3D. This allows the user to see the same information in multiple contexts, and not have to update each one each time.
  • When the user mouses over the area in the second image, their cursor changes to a cross. This is used to indicate that a click and drag will create a new view, and is meant to draw in users, and teach them new functionality.

The bad:

  • Controls are not standard with other applications, even similar ones. Right click is the default select button, but this can be remapped.
  • User on-boarding is limited. Most of the functionality is hidden with keyboard shortcuts, and it is difficult for a user to know how to do a task they have in mind, or even what it is called.

Example 2: MA Jury Duty

Purpose: The website for MA courts, and for responding to jury duty summons.

The good:

  • The most common reason to go to the site is in response to a jury duty summons. Therefore, on the main page of the site, there is a large button to respond to the summons.
  • The language is clear, and tells the user what each step requires, and the choices available.
  • The first page says that the form can be completed through the mail instead,
  • For each page, the survey form is in a central box, which frames the important information and separates it from the site.

The bad:

  • The arrangement of text occasionally puts more emphasis on less important information.

Example 3: Rayman 3 Menu

Purpose: Rayman 3 is a video game; this is regarding its pause menu.

The good:

  • The main menu is standard: Options, Load, etc. This is the screen that the menu defaults to, since it is likely the most used one. Therefore, the most common functionality is the easiest to access.

The bad:

  • The controls are completely non-standard on the platform, and are clumsy. There is no mouse control for the menu. The left and right control keys move between menus, and backspace is resume. Further, the controls for these functions cannot be altered.
  • There is no indication of where a particular functionality is, so the user simply has to scroll through 4-10 menus looking for it.
  • The absence of a usual feature (a manual button) is not mentioned or explained. This is not addressed in the game proper either.

Example 4: Eclipse

Purpose: Eclipse is an IDE for several programming languages, e.g. Java.

The good:

  • The menus are searchable, which means if the user knows the name, it's easier to find. The search also looks in text within menus, not just titles.

The bad:

  • Functionality is often hidden in submenus, or hard to access menus. Adjusting font size or colour requires right clicking on the gutter of the editor, as seen in the second picture above.
  • Names are often non-standard or non-consistent. In Project->Properties, there is a menu called Resource. In it are things like path or environment variables. Those are technically resources, but a more specific name would make navigating the menus easier.
